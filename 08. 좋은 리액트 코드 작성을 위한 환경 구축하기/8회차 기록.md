## 논의

**ESLint**

- ts의 enum은 트리쉐이킹 적용되지 않으니 주의가 필요하다.
- stylelint → css를 린트해준다!
- husky를 사용하면 커밋 전에 린트를 확인할 수 있다.
- git action에 lint를 걸어서 pr 단계에서 확인할 수 있다. 이런 방법이 우회가 불가능해서 더 좋은 것 같다.
- import react → 리액트 17부터는 안해도 된다!

**Testing**

- 테스트는 취약한 부분부터 시작해야한다.
- 테스트를 작성하는지? → 백엔드 보다는 범위가 넓은 것 같다. 유틸 함수 레벨에서는 작성한다.
- 컴포넌트 테스트에 스토리 북 많이 사용 → 해야할게 너무 많다. 문서화에는 유리하지만 구축하는게 어렵다.
- 프론트엔드 TDD → [프론트엔드 TDD](https://youtu.be/L1dtkLeIz-M?si=xtnJ8ZT6MalHP4TF)
- 테스트 코드는 제대로 된 회사면 있다. → 어떻게 만드는지는 알아야 할 것 같다.
- 테스트의 위치는? → 같은 폴더에 둔다. 일종의 문서화 역할도 있다고 생각한다.

**8장 이외의 내용들**

- 삼항 연산자가 return에 있는 경우 어떤걸로 대체할 수 있을까? → 얼리 리턴으로 대체한다. [얼리 리턴](https://medium.com/swlh/return-early-pattern-3d18a41bba8)
- 개발 할 때 아키텍처를 짜고 시작하는지? → FSD가 유행하는 것 같다. [FSD 아키텍처](https://velog.io/@yeonoey/FSD-%EC%95%84%ED%82%A4%ED%85%8D%EC%B3%90) | [FSD 관점으로 코드 경계 찾기](https://velog.io/@teo/fsd)
- css 스타일 관리 어떻게 하는지? → 테일 윈드 사용, 스타일드 컴포넌트, [바닐라 익스트렉트](https://velog.io/@keumky1/Vanilla-Extract%EB%9E%80-%EB%AC%B4%EC%97%87%EC%9D%B8%EA%B0%80), 판다css, BEM 네이밍 컨벤션
- 스타일드 컴포넌트 + NEXT → 서버 컴포넌트에는 적용이 안되나요? → 이모션은 불가능하지만 스타일드 컴포넌트는 가능하다.
